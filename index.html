    Appendix A.1.1 <accentuationPattern>

    <accentuationPattern> (accentuation pattern) A metrical accentuation instruction specifies which accentuation pattern should be applied at and after its date.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.reference.name (@name.ref) att.loop (@loop) att.id (@xml:id)
    scale(accentuation scaling factor) This attribute is required to scale the pattern's accentuation values (which are defined in [0.0; 1.0]) to actual loudness values.
    StatusRequired
    Datatypedouble
    Note

    Typically, the value is non-negative. It sets the value of the strongest accentuation. Its negative marks the strongest restraint. Setting scale to a negative value will invert the accentuation pattern. Setting scale to 0.0 effectively deactivates the accentuation.

    stickToMeasures(stick to measures flag) Set this attribute true if the accentuation pattern should restart at the beginning of each measure.
    StatusOptional
    Datatypeboolean
    Note

    Accentuation patterns do not need to have the same length as a measure. Thus,its beginning does not necessarily coincide with the beginning of a measure. In other words, the first beat in the accentuation pattern is not necessarily the first beat of the measure, same for all further beats. This behaviour is essential to model, for instance, hemiolas. However, if the accentuation pattern is ment to run synchonously with the measures, this attribute should be set true.

    Note

    Default value is true.

    Contained by
    mpm.core: metricalAccentuationMap
    May containEmpty element
    Note

    If the accentuation pattern is looping, it will be repeatedly applied starting at date until the date of the subsequent <accentuationPattern> element.

    ExampleA minimal and a complete accentuation pattern instruction. Note that, according to the values of scale, the former is less pronounced than the latter.
    <accentuationPattern date="0.0"  name.ref="my accentuation pattern 1" scale="6.0"/> <accentuationPattern date="165000.0"  loop="true" name.ref="my accentuation pattern 2"  scale="25.0" stickToMeasures="true"/>
    Note also that a <style> element must be added before the first accentuation pattern instruction and it must refer to a metrical accentuation style with the two patterns in attribute name.ref. Otherwise, "my accentuation pattern 1" and "my accentuation pattern 2" are undefined.

    Appendix A.1.4 <asynchrony>

    <asynchrony> (asynchrony) This element sets a constant milliseconds offset starting a the musical position specified via attribute date ans lasting until the subsequent <asynchrony> element in the <asynchronyMap> specifies a different offset.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.time.physical.milliseconds.offset (@milliseconds.offset) att.id (@xml:id)
    Contained by
    mpm.core: asynchronyMap
    May containEmpty element
    Note

    A negative offset at the beginning that shifts the physical date (milliseconds date) below 0.0 will take no effect in most applications. Take, e.g., a media player; how can it play a note before the user hit the play button? Hence, a negative offset should be avoided if it evoke such a situation.

    Appendix A.1.5 <asynchronyMap>

    <asynchronyMap> (asynchrony map) The <asynchronyMap> is used to specify timing offsets/delays of a musical part or group of parts.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.id (@xml:id)
    Member of
    model.mapLike
    Contained by
    mpm.core: dated
    May contain
    mpm.core: asynchrony
    Note

    Asynchrony should be used to model timing offsets between musical parts that remain constant for a certain time. It is not very handy (even though possible) for fine-grained offset changes, e.g. in terms of continuously varying imprecision in the interaction between the musicians. Therefore, <imprecisionMap.timing> should be used. If the variations are of a more systematic fashion, <rubatoMap> is probably a better tool. All these micro-timing maps can, of course, be used in combination to create more complex effects, e.g. a swinging (via <rubatoMap>) laid-back (via <asynchronyMap>) double bass with a bit of random imprecision (via <imprecisionMap.timing>).

    ExampleWith the following <asynchronyMap> the musical part starts with a delay of 100 milliseconds. Later the offset changes to -50 milliseconds, i.e. the part comes 50 milliseconds early.
    <asnchronyMap>  <asynchrony date="0.0"   milliseconds.offset="100.0"/>  <asynchrony date="34650.0"   milliseconds.offset="-50.0"/> </asnchronyMap>

    Appendix A.1.7 <distribution.correlated.brownianNoise>

    <distribution.correlated.brownianNoise> (Brownian noise distribution) This defines a correlated distribution based on Brownian noise.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.distribution.limits (@limit.lower, @limit.upper) att.time.physical.milliseconds.timingBasis (@milliseconds.timingBasis) att.id (@xml:id)
    stepWidth.max(maximum step width) specifies the maximum value change width from one time step to the next.
    StatusRequired
    Datatypedouble
    Member of
    model.distributionLike
    Contained by
    mpm.core: imprecisionMap imprecisionMap.dynamics imprecisionMap.timing imprecisionMap.toneduration imprecisionMap.tuning
    May containEmpty element
    Note

    An intuitive explanation of Brownian noise in MPM is the random walk in 1D. Suppose at time position t we have a random value x. The next time position is at t+milliseconds.timingBasis. The value at this position is determined by making a random step starting at x. The step width is delimited by stepWidth.max. Thus, possible values are in the interval [x-stepWidth.max; x+stepWidth.max] and further delimited by limit.lower and limit.upper. The fact that the value range at any iteration depends on the values of previous iterations makes this distribution a correlated distribution.

    Example
    <distribution.correlated.brownianNoise date="0.0"  limit.lower="-40.0" limit.upper="40.0"  milliseconds.timingBasis="300.0" stepWidth.max="10.0"/>

    Appendix A.1.10 <distribution.list>

    <distribution.list> (distribution list) This defines a list of distribution values.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.time.physical.milliseconds.timingBasis (@milliseconds.timingBasis) att.id (@xml:id)
    Member of
    model.distributionLike
    Contained by
    mpm.core: imprecisionMap imprecisionMap.dynamics imprecisionMap.timing imprecisionMap.toneduration imprecisionMap.tuning
    May contain
    mpm.core: measurement
    Note

    Distribution lists are deterministic alternatives to stochastic distributions. Instead of random values a distribution list contains a fixed sequence of values that can be applied in any type of imprecision map.

    Note

    A typical scenario for the usage of a distribution list is the following. During timing analysis of a music recording we already created a global <tempoMap> and a local <rubatoMap> for a musician. But there are still timing offsets left that we cannot (yet) explain systematically. We put these offsets into a distribution list for further analysis and to reproduce them during performance rendering. It is important to set the timing basis (the milliseconds length of one measurement sample) as fine as necessary as these lists can become very long.

    ExampleThis example shows a distribution list that captures dynamics fluctuations.
    <imprecisionMap.dynamics>  <distribution.list date="0.0"   timingBasis.milliseconds="300.0">   <measurement value="2.7"/>   <measurement value="6.67"/>   <measurement value="10.45"/>   <measurement value="-7.13"/>   <measurement value="3.2"/>   <measurement value="-2.63"/>   <measurement value="-8.4"/>   <measurement value="6.0"/> <!-- ... -->  </distribution.list> </imprecisionMap.dynamics>

    <dynamics> (dynamics) A dynamics instruction
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.transition.to (@transition.to) att.id (@xml:id)
    volume(volume) The initial volume of this instruction.
    StatusRequired
    Datatypestring
    Note

    The value can be purely numeric. Alternatively, the value can also be literal and, thus, refer to a <dynamicsDef> in the underlying dynamics style definition.

    StatusOptional
    Datatypedouble
    Note

    If this attribute is not specified, the default value is 0.0.

    protraction(asymmetry of the dynamics transition) If attribute transition.to is specified and, thus, the element describes a continuous dynamics transition (cresc., decresc.) the shape of the transition can be further refined by this attribute. Its value in the interval [-1.0; 1.0] indicates whether the transition comes relatively early (value < 0.0), relatively late (value > 0.0) or is equally shared between first and second half of the interval between <date> and the subsequent dynamics instruction's<date>.
    StatusOptional
    Datatypedouble
    Note

    If this attribute is not specified, the default value is 0.0.

    subNoteDynamics(sub-note dynamics flag) If attribute transition.to is specified and, thus, the element describes a continuous dynamics transition (cresc., decresc.) this attribute can be used to indicate whether the dynamics are performed note-wise (false) or on the sub-note level (true).
    StatusOptional
    Datatypeboolean
    Note

    mpm.core: dynamicsMap
    May containEmpty element

    Appendix A.1.16 <global>

    <global> (global performance data) This element includes all information that apply to all parts of a musical performance.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Contained by
    mpm.core: performance
    May contain
    mpm.core: dated header
    Note

    The interplay between global and local (i.e. part-specific) information can be understood as follows. A global map (e.g. <tempoMap>) is applied to each part as long as it does not define a local map of the same type. Even if that local map is empty, its presence signals that the part has its own map and will, thus, ignore the global map. A typical scenario is that the <tempoMap> is defined globally, i.e. all parts follow the same tempo, "follow the same conductor" and stay synchronised. Each part, however, can add individual timing details via a local <rubatoMap>, <asynchronyMap>, <imprecisionMap.timing> etc. The same rule applies to global vs. local style definitions. A part prefers its local information over the global ones.

    Appendix A.3.8 att.pulsesPerQuarter

    att.pulsesPerQuarter (pulsesPerQuarter attribute class) This attribute class provides attribute pulsesPerQuarter.
    Modulempm.midi
    Membersperformance
    Attributes
    pulsesPerQuarter(pulses per quarter note) This attribute specifies the timing resolution of the maps, i.e. of attributes with a symbolic time value such as date and duration.
    StatusRequired
    DatatypenonNegativeInteger
    Note

    Following the convention established by the MIDI standard and step sequencers, we distinguish between two concepts of time in MPM, physical time and symbolic time. Physical time is measured in minutes, seconds, milliseconds etc. Typical attributes using physical time values are absoluteDelayMs and absoluteDurationChangeMs. Symbolic time, by contrast, represents the temporal order of events (such as notes, rests, performance instructions) in musical terms. In symbolic time the musical note values and time positions are indifferent of the underlying tempo, e.g. a quarter note has always the same duration even though the physical duration varies when played in a different tempo. The pulses per quarter note (PPQ) unit, also known as ticks per quarter (TPQ) specifies the smalles integer unit of symbolic time, i.e. the integer resolution of the timing grid.
