      Appendix A.1.4 <global>

      <global> (global performance data) This element includes all information that apply to all parts of a musical performance.
      Namespacehttp://www.cemfi.de/mpm/ns/1.0
      Modulempm.core
      Contained by
      mpm.core: performance
      May contain
      mpm.core: dated header
      Note

      The interplay between global and local (i.e. part-specific) information can be understood as follows. A global map (e.g. <tempoMap>) is applied to each part as long as it does not define a local map of the same type. Even if that local map is empty, its presence signals that the part has its own map and will, thus, ignore the global map. A typical scenario is that the <tempoMap> is defined globally, i.e. all parts follow the same tempo, "follow the same conductor" and stay synchronised. Each part, however, can add individual timing details via a local <rubatoMap>, <asynchronyMap>, <imprecisionMap.timing> etc. The same rule applies to global vs. local style definitions. A part prefers its local information over the global ones.

      ExampleThis is the typical structure of a <performance>.
      <performance name="a performance"  pulsesPerQuarter="720">  <global>   <header> <!-- header information for all parts -->   </header>   <dated> <!-- performance maps (tempoMap, dynamicsMap etc.) that apply to all parts -->   </dated>  </global>  <part name="Soprano" number="1"   midi.channel="0" midi.port="0">   <header> <!-- header information for this part -->   </header>   <dated> <!-- performance maps that apply to this part -->   </dated>  </part> <!-- further parts --> </performance>
      Content model
      <content autoPrefix="true">
 <classRef key="model.partLike"
  expand="sequence" minOccurs="1" maxOccurs="1"/>
</content>
      Schema Declaration
      element global { mpm.header, mpm.dated }

      Appendix A.1.5 <header>

      <header> (header information) The <header> environment in MPM is the place for non-dateable information, i.e. information that are not tied to a particular position on the time axis, such as style definitions.
      Namespacehttp://www.cemfi.de/mpm/ns/1.0
      Modulempm.core
      Member of
      model.partLike
      Contained by
      mpm.core: global part
      May contain
      mpm.core: articulationStyles dynamicsStyles metricalAccentuationStyles rubatoStyles tempoStyles
      Note

      Each style type is allowed only once in the <header> environment. There cannot be, e.g., more than one <tempoStyles> in the <header>.

      ExampleA typical <header> looks like this.
      <header>  <tempoStyles>   <styleDef name="Baumgartner">    <tempoDef name="Allegro" value="147.0"/>    <tempoDef name="Andante" value="100.0"/>   </styleDef>  </tempoStyles>  <dynamicsStyles>   <styleDef name="Baumgartner">    <dynamicsDef name="p" value="48.0"/>    <dynamicsDef name="f" value="97.0"/>    <dynamicsDef name="sf" value="127.0"/>   </styleDef>  </dynamicsStyles>  <articulationStyles>   <styleDef name="Baumgartner">    <articulationDef name="nonlegato"     relativeDuration="0.95"/>    <articulationDef name="stacc"     absoluteDurationMs="160.0" absoluteVelocityChange="-5.0"/>    <articulationDef name="legato"     relativeDuration="1.0"/>    <articulationDef name="legatoStop"     relativeDuration="0.8" relativeVelocity="0.7"/>    <articulationDef name="spicc"     absoluteDurationMs="140.0" absoluteVelocityChange="25.0"/>   </styleDef>  </articulationStyles> </header>
      Content model
      <content autoPrefix="true">
 <classRef key="model.headerLike"
  expand="sequenceOptional" minOccurs="0" maxOccurs="1"/>
</content>

      <mpm> (MPM document) The root element of an MPM-conformant document.
      Namespacehttp://www.cemfi.de/mpm/ns/1.0

      ExampleThis is a minimal MPM document.
      <mpm>  <performance name="a performance"   pulsesPerQuarter="720">   <global>    <header/>    <dated/>   </global>  </performance> </mpm>

      Appendix A.1.9 <performance>

      <performance> (performance description) This element includes all information of a single musical performance.
      Namespacehttp://www.cemfi.de/mpm/ns/1.0
      Modulempm.core
      Attributesatt.name (@name) att.pulsesPerQuarter (@pulsesPerQuarter)
      Contained by
      mpm.core: mpm
      May contain
      mpm.core: global part
      Note

      In the context of the Music Performance Markup format a musical performance description comprises all information necessary to transform symbolic music data (e.g. sheet music represented as MEI or MSM) into a representation (MIDI) that can be played back by any kind of sound generator (e.g. synthesizer or sampler) and, thus, be synthesised to audio. While the symbolic music data may already be playable via a trivial transformation (similar to an empty <performance>) the result will sound very mechanic and non-human. With a performance description the performance transformation adds all the details that are require for an expressive interpretation of the symbolic raw data, such as tempo variations, micro timing, dynamics, metrical accentuation, and articulation. This <performance> element provides these information in a formalised way that is also accessible to analytical, creative and scientific application domains. Obviously, there can be more than one ways to perform a piece of music. Hence, an MPM can have more than one <performance>. It is even possible to apply a <performance> to multiple pieces of music to experiment how it transfers to a different music context.

      ExampleThis is the typical structure of a <performance>.
      <performance name="a performance"  pulsesPerQuarter="720">  <global>   <header> <!-- header information for all parts -->   </header>   <dated> <!-- performance maps (tempoMap, dynamicsMap etc.) that apply to all parts -->   </dated>  </global>  <part name="Soprano" number="1"   midi.channel="0" midi.port="0">   <header> <!-- header information for this part -->   </header>   <dated> <!-- performance maps that apply to this part -->   </dated>  </part> <!-- further parts --> </performance>
      Content model
      <content autoPrefix="true">
 <elementRef key="global" minOccurs="1"
  maxOccurs="1"/>
 <elementRef key="part" minOccurs="0"
  maxOccurs="unbounded"/>
</content>
      Schema Declaration
      element performance
{
   mpm.att.name.attributes,
   mpm.att.pulsesPerQuarter.attributes,
   mpm.global,
   mpm.part*
}

      Appendix A.1.10 <relatedResources>

      <relatedResources> (related resources) This element can be used to proved references to related resources of (musical) data.
      Namespacehttp://www.cemfi.de/mpm/ns/1.0
      Modulempm.resources
      Contained by
      mpm.core: mpm
      May contain
      mpm.resources: resource
      Note

      This list of resource references is basically of an informal nature and does not imply any specific meaning. E.g., performances are not restricted to be applied only to the MSM objects listed here. It is up to the user/application to be more specific with this, e.g. by means of edition guidelines for a historical-critical music edition.

      ExampleThe element <relatedResources> holds one or more references to other resources, e.g. the MEI file from which it was derived or an MSM file which it is intended to be applied to.
      <relatedResources>  <resource uri="pathTo/music.mei"   type="mei"/>  <resource uri="pathTo/music.msm"   type="msm"/>  <resource uri="pathTo/music.mid"   type="midi"/>  <resource uri="pathTo/another.mid"   type="midi"/>  <resource uri="pathTo/music.xml"   type="musicXML"/>  <resource uri="pathTo/music.mxl"   type="musicXML"/>  <resource uri="pathTo/unsupportedMusicType.umt"   type="umt"/> </relatedResources>

      att.midiRouting (MIDI routing attributes class) This attribute class provides attributes midi.channel and midi.port.
      Modulempm.midi
      Memberspart
      Attributes
      midi.channel(MIDI channel number) Channel numbering starts with 0! The first MIDI channel is 0. the 16th MIDI channel is 15.
      StatusRequired
      DatatypenonNegativeInteger
      midi.port(MIDI port number) Each MIDI device (MIDI port) can have up to 16 channels (i.e. voices) which is sometimes not enough, e.g. for an orchestral setup. To address further voices choose another port.

      Appendix A.3.4 att.pulsesPerQuarter

      att.pulsesPerQuarter (pulsesPerQuarter attribute class) This attribute class provides attribute pulsesPerQuarter.
      Modulempm.midi
      Membersperformance
      Attributes
      pulsesPerQuarter(pulses per quarter note) This attribute specifies the timing resolution of the maps, i.e. of attributes with a symbolic time value such as date and duration.
      StatusRequired
      DatatypenonNegativeInteger
      Note

      Following the convention established by the MIDI standard and step sequencers, we distinguish between two concepts of time in MPM, physical time and symbolic time. Physical time is measured in minutes, seconds, milliseconds etc. Typical attributes using physical time values are absoluteDelayMs and absoluteDurationChangeMs. Symbolic time, by contrast, represents the temporal order of events (such as notes, rests, performance instructions) in musical terms. In symbolic time the musical note values and time positions are indifferent of the underlying tempo, e.g. a quarter note has always the same duration even though the physical duration varies when played in a different tempo. The pulses per quarter note (PPQ) unit, also known as ticks per quarter (TPQ) specifies the smalles integer unit of symbolic time, i.e. the integer resolution of the timing grid.
