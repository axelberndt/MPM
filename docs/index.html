    Appendix A.1.3 <asynchrony>

    <asynchrony> (asynchrony) This element sets a constant milliseconds offset starting a the musical position specified via attribute date ans lasting until the subsequent <asynchrony> element in the <asynchronyMap> specifies a different offset.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.time.physical.milliseconds.offset (@milliseconds.offset) att.id (@xml:id)
    Contained by
    mpm.core: asynchronyMap
    May containEmpty element
    Note

    A negative offset at the beginning that shifts the physical date (milliseconds date) below 0.0 will take no effect in most applications. Take, e.g., a media player; how can it play a note before the user hit the play button? Hence, a negative offset should be avoided if it evoke such a situation.

    ExampleWith the following <asynchronyMap> the musical part starts with a delay of 100 milliseconds. Later the offset changes to -50 milliseconds, i.e. the part comes 50 milliseconds early.
    <asnchronyMap>  <asynchrony date="0.0"   milliseconds.offset="100.0"/>  <asynchrony date="34650.0"   milliseconds.offset="-50.0"/> </asnchronyMap>

 <empty/>
</content>

    element asynchrony

}

    Appendix A.1.4 <asynchronyMap>

    <asynchronyMap> (asynchrony map) The <asynchronyMap> is used to specify timing offsets/delays of a musical part or group of parts.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.id (@xml:id)
    Member of
    model.mapLike
    Contained by
    mpm.core: dated
    May contain
    mpm.core: asynchrony
    Note

    Asynchrony should be used to model timing offsets between musical parts that remain constant for a certain time. It is not very handy (even though possible) for fine-grained offset changes, e.g. in terms of continuously varying imprecision in the interaction between the musicians. Therefore, <imprecisionMap.timing> should be used. If the variations are of a more systematic fashion, <rubatoMap> is probably a better tool. All these micro-timing maps can, of course, be used in combination to create more complex effects, e.g. a swinging (via <rubatoMap>) laid-back (via <asynchronyMap>) double bass with a bit of random imprecision (via <imprecisionMap.timing>).

    ExampleWith the following <asynchronyMap> the musical part starts with a delay of 100 milliseconds. Later the offset changes to -50 milliseconds, i.e. the part comes 50 milliseconds early.
    <asnchronyMap>  <asynchrony date="0.0"   milliseconds.offset="100.0"/>  <asynchrony date="34650.0"   milliseconds.offset="-50.0"/> </asnchronyMap>

    <content autoPrefix="true">

  maxOccurs="unbounded"/>
</content>

    element asynchronyMap { mpm.att.id.attributes, mpm.asynchrony* }

    Appendix A.1.5 <dated>

    <dated> (dated information) The <dated> environment is the place for all datable information, ie. that are located at a specific position in time. They are organised in maps. Each map represents a specific domain, such as tempo, rubato, asynchrony, dynamics, metrical accentuation, articulation, etc.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Member of
    model.partContent
    Contained by
    mpm.core: global part
    May contain
    mpm.core: articulationMap asynchronyMap dynamicsMap imprecisionMap imprecisionMap.dynamics imprecisionMap.timing imprecisionMap.toneduration imprecisionMap.tuning metricalAccentuationMap rubatoMap tempoMap
    Note

    Each type of map is allowed only once in the <dated> environment. There cannot be, e.g., more than one <tempoMap> in the <dated>.

    ExampleThe following <dated> element holds two maps, one for tempo instructions, the other for dynamics instructions.

    Appendix A.1.6 <distribution.correlated.brownianNoise>

    <distribution.correlated.brownianNoise> (Brownian noise distribution) This defines a correlated distribution based on Brownian noise.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.time.symbolic.date (@date) att.distribution.limits (@limit.lower, @limit.upper) att.time.physical.milliseconds.timingBasis (@milliseconds.timingBasis) att.id (@xml:id)
    stepWidth.max(maximum step width) specifies the maximum value change width from one time step to the next.
    StatusRequired
    Datatypedouble
    Member of
    model.distributionLike
    Contained by
    mpm.core: imprecisionMap imprecisionMap.dynamics imprecisionMap.timing imprecisionMap.toneduration imprecisionMap.tuning
    May containEmpty element
    Note

    An intuitive explanation of Brownian noise in MPM is the random walk in 1D. Suppose at time position t we have a random value x. The next time position is at t+milliseconds.timingBasis. The value at this position is determined by making a random step starting at x. The step width is delimited by stepWidth.max. Thus, possible values are in the interval [x-stepWidth.max; x+stepWidth.max] and further delimited by limit.lower and limit.upper. The fact that the value range at any iteration depends on the values of previous iterations makes this distribution a correlated distribution.

    Example
    <distribution.correlated.brownianNoise date="0.0"  stepWidth.max="10.0" limit.lower="-40.0" limit.upper="40.0"  milliseconds.timingBasis="300.0"/>
    Content model
    <content autoPrefix="true">
 <empty/>
</content>

    Appendix A.1.13 <global>

    <global> (global performance data) This element includes all information that apply to all parts of a musical performance.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Contained by
    mpm.core: performance
    May contain
    mpm.core: dated header
    Note

    The interplay between global and local (i.e. part-specific) information can be understood as follows. A global map (e.g. <tempoMap>) is applied to each part as long as it does not define a local map of the same type. Even if that local map is empty, its presence signals that the part has its own map and will, thus, ignore the global map. A typical scenario is that the <tempoMap> is defined globally, i.e. all parts follow the same tempo, "follow the same conductor" and stay synchronised. Each part, however, can add individual timing details via a local <rubatoMap>, <asynchronyMap>, <imprecisionMap.timing> etc. The same rule applies to global vs. local style definitions. A part prefers its local information over the global ones.

    ExampleThis is the typical structure of a <performance>.
    <performance name="a performance"  pulsesPerQuarter="720">  <global>   <header> <!-- header information for all parts -->   </header>   <dated> <!-- performance maps (tempoMap, dynamicsMap etc.) that apply to all parts -->   </dated>  </global>  <part name="Soprano" number="1"   midi.channel="0" midi.port="0">   <header> <!-- header information for this part -->   </header>   <dated> <!-- performance maps that apply to this part -->   </dated>  </part> <!-- further parts --> </performance>
    Content model
    <content autoPrefix="true">
 <classRef key="model.partContent"
  expand="sequence" minOccurs="1" maxOccurs="1"/>
</content>
    Schema Declaration
    element global { mpm.header, mpm.dated }

    Appendix A.1.16 <imprecisionMap.dynamics>

    <imprecisionMap.dynamics> (imprecision map dynamics) This map is used to model random dynamics fluctuations.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.id (@xml:id)
    Member of
    model.mapLike
    Contained by
    mpm.core: dated
    May contain
    mpm.core: distribution.correlated.brownianNoise distribution.correlated.compensatingTriangle distribution.gaussian distribution.triangular distribution.uniform
    Note

    Editors should keep in mind that the dynamics fluctuations are additive. At the end of the dynamics computation pipeline each musical event has a certain dynamics value. The distribution function applied in the <imprecisionMap.dynamics> adds a random offset to this. Hence, the distribution values are interpreted as relate (no absolute) dynamics values.

    <imprecisionMap.dynamics>  <distribution.gaussian date="0.0"   deviation.standard="7.5" limit.lower="-15.0" limit.upper="15.0"/> </imprecisionMap.dynamics>

    Appendix A.1.18 <imprecisionMap.toneduration>

    <imprecisionMap.toneduration> (imprecision map tone duration) This map is used to model random imprecision in articulation/tone duration.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.id (@xml:id)
    Member of
    model.mapLike
    Contained by
    mpm.core: dated
    May contain
    mpm.core: distribution.correlated.brownianNoise distribution.correlated.compensatingTriangle distribution.gaussian distribution.triangular distribution.uniform
    Note

    Editors should keep in mind that the random variations that this imprecision map generates are additive. The values do not represent absolute tone durations but changes of already given tone durations. Hence, this does not overwrite articulation but adds variation to it.     Appendix A.1.23 <part>

    <part> (part-specific performance data) This element includes all information that applies to a specific musical part or voice.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.name (@name) att.number (@number) att.id (@xml:id) att.midiRouting (@midi.channel, @midi.port)
    Contained by
    mpm.core: performance
    May contain
    mpm.core: dated header
    Note

    Basically, a part in MPM can also be polyphonic, i.e. it represents a collection of parts. E.g., in orchestral music notation one staff may contain two parts. Thus, the performance instructions notated in the staff apply to both.

    Note

    The interplay between global and local (i.e. part-specific) information can be understood as follows. A global map (e.g. <tempoMap>) is applied to each part as long as it does not define a local map of the same type. Even if that local map is empty, its presence signals that the part has its own map and will, thus, ignore the global map. A typical scenario is that the <tempoMap> is defined globally, i.e. all parts follow the same tempo, "follow the same conductor" and stay synchronised. Each part, however, can add individual timing details via a local <rubatoMap>, <asynchronyMap>, <imprecisionMap.timing> etc. The same rule applies to global vs. local style definitions. A part prefers its local information over the global ones.

    ExampleThis is the typical structure of a <performance>.
    <performance name="a performance"  pulsesPerQuarter="720">  <global>   <header> <!-- header information for all parts -->   </header>   <dated> <!-- performance maps (tempoMap, dynamicsMap etc.) that apply to all parts -->   </dated>  </global>  <part name="Soprano" number="1"   midi.channel="0" midi.port="0">   <header> <!-- header information for this part -->   </header>   <dated> <!-- performance maps that apply to this part -->   </dated>  </part> <!-- further parts --> </performance>

    Appendix A.1.24 <performance>

    <performance> (performance description) This element includes all information of a single musical performance.
    Namespacehttp://www.cemfi.de/mpm/ns/1.0
    Modulempm.core
    Attributesatt.name (@name) att.id (@xml:id) att.pulsesPerQuarter (@pulsesPerQuarter)
    Contained by
    mpm.core: mpm
    May contain
    mpm.core: global part
    Note

    In the context of the Music Performance Markup format a musical performance description comprises all information necessary to transform symbolic music data (e.g. sheet music represented as MEI or MSM) into a representation (MIDI) that can be played back by any kind of sound generator (e.g. synthesizer or sampler) and, thus, be synthesised to audio. While the symbolic music data may already be playable via a trivial transformation (similar to an empty <performance>) the result will sound very mechanic and non-human. With a performance description the performance transformation adds all the details that are require for an expressive interpretation of the symbolic raw data, such as tempo variations, micro timing, dynamics, metrical accentuation, and articulation. This <performance> element provides these information in a formalised way that is also accessible to analytical, creative and scientific application domains. Obviously, there can be more than one ways to perform a piece of music. Hence, an MPM can have more than one <performance>. It is even possible to apply a <performance> to multiple pieces of music to experiment how it transfers to a different music context.

    ExampleThis is the typical structure of a <performance>.
    <performance name="a performance"  pulsesPerQuarter="720">  <global>   <header> <!-- header information for all parts -->   </header>   <dated> <!-- performance maps (tempoMap, dynamicsMap etc.) that apply to all parts -->   </dated>  </global>  <part name="Soprano" number="1"   midi.channel="0" midi.port="0">   <header> <!-- header information for this part -->   </header>   <dated> <!-- performance maps that apply to this part -->   </dated>  </part> <!-- further parts --> </performance>

    Appendix A.3.7 att.pulsesPerQuarter

    att.pulsesPerQuarter (pulsesPerQuarter attribute class) This attribute class provides attribute pulsesPerQuarter.
    Modulempm.midi
    Membersperformance
    Attributes
    pulsesPerQuarter(pulses per quarter note) This attribute specifies the timing resolution of the maps, i.e. of attributes with a symbolic time value such as date and duration.
    StatusRequired
    DatatypenonNegativeInteger
    Note

    Following the convention established by the MIDI standard and step sequencers, we distinguish between two concepts of time in MPM, physical time and symbolic time. Physical time is measured in minutes, seconds, milliseconds etc. Typical attributes using physical time values are absoluteDelayMs and absoluteDurationChangeMs. Symbolic time, by contrast, represents the temporal order of events (such as notes, rests, performance instructions) in musical terms. In symbolic time the musical note values and time positions are indifferent of the underlying tempo, e.g. a quarter note has always the same duration even though the physical duration varies when played in a different tempo. The pulses per quarter note (PPQ) unit, also known as ticks per quarter (TPQ) specifies the smalles integer unit of symbolic time, i.e. the integer resolution of the timing grid.
